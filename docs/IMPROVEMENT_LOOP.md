# 小さくリッチにし続ける改善ループ

このサイトは一度の大改修ではなく、短いルーチンで「読み味」「触り味」「情報の濃さ」を増やしていく。

## 1回の作業単位

現在の定期改善は1時間ごとのサイクルを前提にする。目安は30分から75分。リッチな演出、セクション間の間奏、スクロール体験、CSS / JS / Astroの境界整理など、確認しながら進めたい作業は90〜120分程度まで広げてよい。

ここでいう「小さく」は、地味にするという意味ではない。焦点がひとつで、戻しやすく、検証できる単位にするという意味。1時間枠では、1つの導線や1つの構造課題に閉じているなら、複数ファイルへまたがる少し大きめの改善も扱ってよい。演出そのものは少し過剰でもよい。

1. 今日の焦点を決める
2. 直近ログと前回差分を見て、同じ場所を雑に重ねていないか確認する
3. 1セクション、またはセクション間の流れを観察する
4. 焦点を絞って変更する
5. 可能ならブラウザ相当で確認する
6. 変更ログに残す

## 焦点の選び方

- Narrative: 文章、順番、余韻をよくする
- Visual: 余白、色、画像、カードの密度をよくする
- Motion: スクロール、粒子、ホバー、ゾーン遷移をよくする
- Content: 制作物、構想、Research Log候補を増やす
- Access: モバイル、可読性、リンク、導線をよくする
- Structure: データ化、命名、ファイル分割、将来のCMS化を進める

## 構造改善の回し方

Structureは毎回やらなくてよい。ただし、Narrative / Visual / Motion の改善が続いたら、数回に一度は構造の掃除を挟む。

候補:

- `src/data/` のグループや命名を、次の編集者が迷わない形に寄せる
- `src/scripts/` の役割境界を見直し、演出の差し替え口を細くする
- `src/pages/index.astro` に残った重複や長い塊を、見た目を変えずに整理する
- `scripts/phase-gate.mjs` に、壊したくない導線や配信前提のチェックを足す
- island化や依存追加は目的化せず、既存vanilla JSの境界整理で足りるかを先に見る

## 作業前チェック

- `RFP.md` の思想から外れていないか
- `RFP.md` の技術案を機械的に実装しようとしていないか。現行のAstro構成と実装状態を優先する。
- `docs/WEB_IMPROVEMENT_PRINCIPLES.md` の「Webでやること / まだWebでやらないこと」から外れていないか
- リッチ化が、情報追加ではなく体験密度・記憶残り・参道の流れを強めているか
- 1ページ内で意味の流れが増えているか
- 演出が本文の読みやすさを邪魔していないか
- 追加するものは `src/data/` に置けないか
- `main` へpushすると https://hazakura-labo-web.pages.dev に自動デプロイされる前提で安全な変更か

## RFPの読み方

`RFP.md` は、葉桜ラボWebサイトの思想・役割・成功条件の正本として扱う。一方で、技術セクションに出てくるディレクトリ例やコンポーネント案は、現行実装を無視して必ず作るものではない。

現在の本線は、Astroの1ページ構造、`src/data/` のセクション別データ、`src/scripts/` のvanilla JS演出、`src/pages/*.js.ts` の互換配信である。新しい構成へ寄せる場合は、見た目と導線を壊さず、1回の作業で検証できる単位にする。1時間サイクルでは、データ境界、描画、スタイル、phase gateを一緒に触った方が安全な場合、同じ焦点に閉じる範囲でまとめて進めてよい。

## 作業後チェック

最低限この3つを見る。

```sh
npm run check:phase
```

オートメーション実行では、macOSのGUI権限やChrome.app起動権限により、Chromium / Chromeの直接起動が失敗することがある。毎回同じ失敗を繰り返さないため、通常の小改善では上記のPhase gateと、必要に応じたNodeデータ検査を標準の完了条件にする。

ブラウザ相当の確認は best-effort とする。使える場合は Playwright など、プロジェクトから再現しやすい方法を優先し、`/Applications/Google Chrome.app` の直接起動は標準手順にしない。ブラウザ起動が権限エラーや無出力で失敗した場合は、短く記録して再試行を重ねず、静的チェックで代替する。

ブラウザ相当で見る場合は以下を見る。

- ファーストビューでブランドと世界観が伝わる
- `構造` セクションが崩れていない
- 制作物カードが増減してもレイアウトが壊れない
- モバイル幅で横スクロールが出ない

## オートメーションで避けること

定期実行では、改善そのものに関係ない環境問題へ長く寄り道しない。

- Chrome.appやPlaywrightなどブラウザ起動が一度失敗したら、同じ実行内で深追いしない。
- ローカルサーバ確認が必要な変更でない限り、サーバを起動したままにしない。起動した場合は最後に停止する。
- 依存追加、フレームワーク導入、大量生成、画像生成、大きな整形は通常ループで行わない。必要なら別タスクとして切り出す。
- ただし、既存構成の範囲で完結するCSS/vanilla JSの演出、余白、トランジション、セクション間の間奏は通常ループで扱ってよい。大事なのは規模ではなく、焦点が絞られていて検証できること。
- `.DS_Store`、スクリーンショット、検査用の一時ファイル、ブラウザプロファイルはコミットしない。
- 未追跡ファイルや既存差分があっても、今回の改善に関係ないものは触らず、コミット対象を明示的に絞る。
- `src/data/` 追記中心の変更では、ブラウザ確認よりもデータ構造のNode検査を優先する。

## デプロイ前チェック

GitHubの `main` にpushするとCloudflare Pagesへ自動デプロイされる。Astro移行後は `npm run build` で `dist/` を生成する前提で確認する。

Astro本線化後も、デプロイ前は次の崩れに注意する。

- Cloudflare Pagesのroot/build/output設定が変わる
- `./img/...` や `downloads/...` の相対パスが変わる
- dev/preview確認とPages上の挙動がずれる
- Canvas、スクロールゾーン、カスタムカーソルの初期化順が変わる
- 生成後のHTML/CSS/JS構造により、現在の1ページ演出が崩れる

公開前は `npm run check:phase` を通し、必要な変更ではローカルpreviewで主要セクションとProjectsフィルターを見る。
