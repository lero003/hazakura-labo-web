# 葉桜ラボ Webサイト改善・拡張 RFP

## AIと自然、思想と制作が循環する、小さな実験室

- **対象サイト**: https://hazakura-labo-web.pages.dev
- **現在の前提**: Astro + Cloudflare Pages による静的Webサイト
- **文書の目的**: 葉桜ラボWebサイトを、現在の活動実態と実装状態に合わせて育てるための判断基準をまとめる。
- **想定読者**: Codex / 実装担当AI / 将来の保守担当 / 葉桜ラボ運営者
- **更新メモ**: 旧RFPの Next.js / React / Tailwind CSS / Framer Motion 前提は廃止。技術案は参考情報として扱い、現行のAstro構成を正とする。

---

## 1. プロジェクト概要

葉桜ラボは、AI、自然、思想、創作、アプリ開発を横断する個人研究・制作ラボである。

Webサイトは、単なる紹介ページではなく、以下が循環して見える「活動母艦」として育てる。

- Kindle出版物
- Webアプリ / ローカルアプリ開発
- AIエージェント / ローカルLLM / 自動化実験
- 哲学的エッセイ・創作物
- 研究ログ・試作ログ
- 将来的なAI万屋 / AI寺子屋 / 地域知識循環構想

目的は、既存の1ページ表現を壊さず、現在の活動実態に合う情報設計と、今後の追加更新に耐える構造へ少しずつ育てること。

## 2. 現在のサイト状況

現行サイトは、Astro 6系の静的サイトとして構築され、Cloudflare Pagesで配信されている。

主な構成:

- Hero
- Philosophy
- One-page Architecture
- Library
- Library -> Projects bridge
- Research Log
- Projects
- Vision
- Quote / 余韻

実装上は、トップページの骨格を `src/pages/index.astro` に置き、表示データを `src/data/`、演出や描画を `src/scripts/`、スタイルを `src/styles/site.css` に分けている。`src/pages/*.js.ts` と `src/pages/style.css.ts` は、Astroから互換的にJS/CSSを配信するためのエンドポイントとして扱う。

既に反映済みの活動実績:

- 『チカちゃんの哲学冒険譚』
- 『重さのないノート』
- Webサイト制作
- アプリ開発
- AI支援ツール構想
- ローカルLLM / エージェント活用実験
- 創作・哲学・エッセイ系コンテンツ

現在の課題:

- 「きれいな思想サイト」に見えすぎると、制作活動・出版実績・実験の厚みが伝わりにくい。
- Library、Projects、Research Log、Visionが同じ循環の中にあることを、より自然に見せたい。
- `src/pages/index.astro`、`src/styles/site.css`、`src/scripts/` は育ってきているため、体験改善と同じくらい構造整理も重要になっている。
- 1ページサイトの美しさを維持しつつ、将来の詳細ページ化やログ増加に耐える境界を育てる必要がある。

## 3. ブランド・フィロソフィー

### 3.1 コア思想

**変化の肯定**

満開の桜だけでなく、花が散り、若葉が混ざる「葉桜」の状態に価値を見出す。完成だけでなく、途中、試行錯誤、未完成、変化の過程そのものを肯定する。

**理と情の融合**

AI、コード、アプリ、論理、構造といった「理」と、自然、生活、物語、直感、情緒といった「情」を対立させず、溶け合わせる。

**道楽としての探求**

効率や収益だけを目的にせず、「面白いから作る」「不思議だから調べる」「生活の中で試す」という道楽的な知性を大切にする。

**知恵の循環**

出版物、アプリ、研究ログ、地域構想、AIツールが互いに循環し、使うほどに知見が蓄積される状態を目指す。

### 3.2 中心コピー

現在の中心コピーは維持する。

> AIと、語ろう。自然と、遊ぼう。

補助説明は、必要に応じて以下の方向へ寄せる。

> 葉桜ラボは、AIと自然、思想と制作、言葉とアプリをつなぐ小さな実験室です。Kindle出版、Webアプリ開発、AIエージェント実験、創作プロジェクトを通じて、変化の中にある知性と遊びを形にしています。

### 3.3 トーン & マナー

- 知的だが、硬すぎない。
- 透明感がある。
- 少し不思議だが、怪しすぎない。
- 個人ラボらしい温度感がある。
- AIを過剰な未来技術として煽らない。
- 自然、生活、哲学、遊びの感覚を大切にする。
- 「研究所」だが、閉じた専門機関ではなく、庭や書庫のように入れる場所にする。

## 4. サイトの役割

葉桜ラボWebサイトは、複数の役割を1ページの中で束ねる。

- **ブランドサイト**: 葉桜ラボとは何かを初見の人に伝える。
- **出版物紹介ページ**: Kindle本がどんな問いから生まれたかを伝える。
- **制作物ポートフォリオ**: アプリ、Webツール、AI実験、ローカルLLM関連の試作物を紹介する。
- **研究ログ**: 完成物だけでなく、問い、実験、失敗、気づきを残す。
- **将来的な入口**: AI万屋、AI寺子屋、自分史、地域AI、レトロガジェット再生などの窓口になる。

現時点では、本格フォーム、認証、投稿管理、CMS、バックエンド運用基盤は作らない。Web上では「入口の気配」や「相談したくなる雰囲気」までを扱う。

## 5. 情報設計

当面は1ページ構成を維持する。主要な流れは以下を基本とする。

1. **Hero**: 一言で空気を伝える。
2. **Philosophy**: 葉桜ラボの根にある思想を示す。
3. **Architecture**: 1ページの中に層があることを示す。
4. **Library**: 出版物を「問いの結晶」として見せる。
5. **Research Log**: 試行錯誤と気づきを置く。
6. **Projects**: 制作物を「問いから生まれた実験」として見せる。
7. **Vision**: AI万屋、AI寺子屋、自分史、地域知識循環などを未完成の構想として正直に示す。
8. **Quote / 余韻**: 問い、実験、成果、次の問いが循環する感覚で閉じる。

将来的に追加できる詳細ページ候補:

- `/library/[slug]`
- `/projects/[slug]`
- `/logs/[slug]`
- `/vision/[slug]`

ただし、最初から全ページを作る必要はない。データ構造とコンポーネント境界だけ、詳細ページ化を妨げない形へ寄せる。

## 6. コンテンツ要件

### Hero

初見で「何のサイトか」を伝える。中心コピー、補助説明、主要導線を持つ。主要導線は Philosophy / Library / Projects のどれかへ自然につながること。

### Library

Kindle出版物を中心に見せる。

必須掲載:

- 『チカちゃんの哲学冒険譚』
- 『重さのないノート』

表示観点:

- 表紙画像
- タイトル
- 短い説明
- どんな問いを扱う本か
- Amazonリンク
- 関連タグやテーマ

Libraryは販売棚ではなく、問いが結晶化した書庫として扱う。

### Research Log

最初は本格的なブログ機能でなくてよい。トップページ上に「最近の実験」として数件表示し、将来的にログ一覧・詳細ページへ拡張する。

ログの基本要素:

- 日付
- タイトル
- 問い
- 実験内容
- 気づき
- 関連プロジェクト

### Projects

制作物を「実績」と「実験」の両方として見せる。

表示観点:

- プロジェクト名
- ステータス
- 種別
- 概要
- 背景・動機
- 関連リンク

ステータス分類の目安:

- `released`: 公開済み
- `in-progress`: 開発中
- `prototype`: 試作中
- `concept`: 構想段階

### Vision

構想は未完成のものとして正直に提示してよい。ただし、抽象論だけで終わらせず、「何をしたいか」が伝わる粒度にする。

含めたい方向:

- AI万屋
- AI寺子屋
- 自分史編纂
- 地域知識循環
- レトロガジェット再生
- ローカルAI拠点

## 7. 技術方針

### 7.1 採用中の本線

- Framework: Astro
- Hosting: Cloudflare Pages
- Runtime: Node.js 22系
- Data: `src/data/*.js`
- Page shell: `src/pages/index.astro`
- Script source: `src/scripts/*.js`
- Script delivery: `src/pages/*.js.ts`
- Styles: `src/styles/site.css` delivered through `src/pages/style.css.ts`

### 7.2 技術案の扱い

新RFPに含まれていた `books.ts`、`projects.ts`、`BaseLayout.astro`、`BookCard`、`ProjectCard`、Content Collections、React等の案は、今後の判断材料として扱う。現行サイトの構造と矛盾する場合は、現実の実装を優先する。

特に、現時点では次を必須にしない。

- Next.jsへの移行
- React / Vue / Svelte の導入
- Tailwind CSS導入
- Framer Motion導入
- フルCMS
- 複雑なバックエンド
- ログイン機能

### 7.3 リファクタリング方針

CSS / JS / Astroの整理は、単なる行数削減ではなく、次回以降の改善が迷子にならない構造にするために行う。

- CSSは、重複したセレクタ、似たユーティリティ、過剰なカスケード、使われていないスタイル、同じ値の散在を小さく統合・削除する。
- JSは、`src/scripts/` の責務境界、初期化順、イベントリスナー、DOM探索、reduced-motion / progressive enhancement の扱いを確認する。
- Astroは、`src/pages/index.astro` が肥大化している場合、見た目を大きく変えない小さな分割やデータ整形の整理を候補にする。
- 大きな全面分割や依存追加は目的化しない。

## 8. デザイン要件

デザインコンセプトは **自然の生命力 × デジタルな知性**。

キーワード:

- 葉桜
- 淡い光
- 余白
- 小さな実験室
- 庭
- 書庫
- ノート
- ガラスのような透明感
- 生活に近いAI

UI方針:

- モバイルファースト。
- 1セクション1メッセージを明確にする。
- カードUIは使いすぎず、余白を保つ。
- 書籍、研究ログ、プロジェクト、構想は視覚的に見分けやすくする。
- ボタンやリンクは控えめだが分かりやすくする。
- アニメーションは軽く、邪魔にならない範囲にする。
- 多少の過剰さや不思議さは歓迎するが、本文の読みやすさを塞がない。

## 10. 改善ロードマップ

### Phase 1: 現状把握と小さなリファクタリング

- 現在のAstro構成を確認する。
- `index.astro`、`site.css`、`src/scripts/` の責務を把握する。
- 直書きされている書籍・プロジェクト情報を洗い出す。
- 不要なコードや重複を整理する。

### Phase 2: 情報設計の更新

- Hero文言を現在の活動実態に合わせて調整する。
- LibraryをKindle本2冊以上に対応した書庫として扱う。
- Projectsを制作物一覧ではなく、問いから生まれた実験として整理する。
- Visionを抽象論だけでなく具体構想に分ける。

### Phase 3: データ境界の整理

- 既存の `src/data/` 分割を尊重する。
- 必要なら Library データの分離を検討する。
- Projects / Research / Vision のslug、status、categoryの契約を強くする。
- コンテンツ追加がHTML直書きだけに偏らないようにする。

### Phase 4: Astro構造の整備

- `src/pages/index.astro` の肥大化箇所を見た目を変えずに分ける。
- `src/pages/*.js.ts` の互換配信と `src/scripts/*.js` の関係を保つ。
- SEO / OGP / favicon / sitemap を確認する。

### Phase 5: 拡張準備

- 詳細ページ化のためのslug設計。
- Research Log追加準備。
- Content Collections導入の要否判断。
- Contact / Request導線の検討。

## 11. Codex向け作業指示

最初に確認するもの:

- `RFP.md`
- `README.md`
- `docs/IMPROVEMENT_LOOP.md`
- `docs/RICHNESS_BACKLOG.md`
- `docs/RICHNESS_LOG.md`
- `docs/WEB_IMPROVEMENT_PRINCIPLES.md`
- `docs/ASTRO_MIGRATION_PLAN.md`
- `src/pages/index.astro`
- `src/data/`
- `src/scripts/`
- `src/styles/site.css`
- `git status --short`

作業方針:

- 1回の改善では、体験上の弱点または保守性を落としている構造を1つ選ぶ。
- 1時間ごとの改善では、観察、実装、検証、ログ更新までを見込む。焦点がひとつなら、文言、データ、CSS / JS / Astro、phase gateをまたぐ少し大きめの作業も許容する。
- 新しいカードやセクション追加は、整理・演出改善・実装整理では解決できない場合だけにする。
- 技術面では現行Astro構成を正とし、RFP内の推奨ディレクトリ例を機械的に作らない。
- 既存アンカー、Projectsフィルター、スクロール体験、Cloudflare Pagesのbuild前提を壊さない。
- 変更後は可能な範囲で `npm run check:phase` と `git diff --check` を実行する。

まだやらなくてよいこと:

- Next.jsへの移行
- CMS導入
- ログイン機能
- 複雑なアニメーションライブラリ
- バックエンド構築
- 決済機能
- 大規模なデザイン全面刷新

## 13. 判断メモ

当面は1ページ構成でよい。ただし、LibraryやProjectsが増えてきたら詳細ページ化する。

Research Logを本格的に増やすなら、Astro Content Collectionsは導入候補。書籍とプロジェクトだけなら、まずは既存の `src/data/*.js` を整える。

React等は現時点では不要。インタラクティブなUIやアニメーションが必要になった場合のみ、Astro Islandsとして部分導入を検討する。

CMSは現時点では不要。MarkdownやTypeScript/JavaScriptデータで更新できるうちは、静的管理を優先する。
